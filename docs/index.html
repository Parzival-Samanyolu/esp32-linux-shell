ESP32 Linux Shell

A Linux-style terminal for a $6 microcontroller — with a camera, an SD card, and three programming languages.

Deneyap WROVER-E ESP32-D0WD-V3 · 240 MHz 8 MB PSRAM @ 80 MHz ESP-IDF · C
connect over Wi-Fi from any OS:
telnet 192.168.1.150 2222

$ whatis

An ESP-IDF firmware that turns the board into a networked, Linux-flavoured terminal. Connect with nc / telnet / PuTTY and it behaves like a real shell — files live on a real SD card, HTTP requests are real, and every system stat comes from live hardware APIs. Then it goes further: a camera, a browser file manager, a text editor, and embedded interpreters for Python, JavaScript and C.

🐚

Linux-style shell

ls, cat, cd, echo, mkdir, rm, df, wget, curl, ping, free, uptime, ifconfig, dmesg, uname, reboot — plus arrow-key command history and a live htop.

📷

Camera (OV2640)

photo — 1600×1200 JPEG to SD. video — records an MJPEG .avi clip. stream — live MJPEG in your browser.

💾

SD storage

FAT32 over SPI. Real files, real directories, real timestamps (NTP-synced clock).

🌐

Browser file manager

files — a web UI to list, preview images, download and upload files to the card.

📝

nano editor

A full-screen text editor over telnet — arrow keys, ^O to save, ^X to exit.

🩺

Diagnostics

selftest (PSRAM/SD/camera/Wi-Fi/heap), stress (2-core CPU + PSRAM), neofetch, ps.

$ run --lang

Three real interpreters embedded in the firmware. Write scripts to the SD card and run them by extension: run script.py · run app.js · run prog.c

python — pocketpy (Python 3 subset)
>>> print(2**64)
18446744073709551616
>>> [i*i for i in range(5)]
[0, 1, 4, 9, 16]
js — Duktape (ES5/ES2015)
js> [1,2,3].map(x => x*x)
1,4,9
js> JSON.stringify({ok:true})
{"ok":true}
cc — PicoC (interpreted C)
// hello.c
int main(){ printf("sum=%d\n",1+2+3); }
→ sum=6
These are compact embedded interpreters, not full CPython/Node/GCC — the ESP32 can't host a real compiler or JVM. They run genuine (subset) Python, JavaScript and C directly on the chip.

Notable internals: each network client runs in its own FreeRTOS task; large buffers use PSRAM; the Python interpreter's static data is relocated to PSRAM via a linker fragment; the JS engine's heap uses custom PSRAM allocators; and the C interpreter ships with a bespoke ESP-IDF platform layer.
Built on the Deneyap WROVER-E with ESP-IDF & PlatformIO.
